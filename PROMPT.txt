这是自己写的照片管理工具,前几年攒的照片散落在 iCloud、Google Photos、各个硬盘和 U 盘里,管理起来很乱,自己写一个统一管理,核心场景是导入/查重/分类/浏览/搜索,纯前端不上传云端,所有数据存本地保护隐私。

需求 1:照片用 Photo 实体来管理,字段包括文件名、大小(字节)、拍摄时间(优先用 EXIF 没就用文件 mtime)、导入时间(入库时打)、所属相册 ID(关联 Album)、标签(数组,多对多)、人脸分组 ID(关联 Person)、缩略图(256x256 的 base64,快速列表用)。

需求 2:相册用 Album 实体来管理,字段包括名(用户可编辑)、封面照片 ID(用第一张或用户指定)、数量(自动统计)。

需求 3:人物用 Person 实体来管理,字段包括代表照片(分组里最清晰那张)、缩略图(同上 base64)、姓名(用户填)、备注(自由文本)。

需求 4:导入支持拖拽和点击两种交互,支持 JPEG/PNG/WebP/HEIC 四种格式,HEIC 走 heic2any 库在浏览器里转码成 JPEG 显示(进度条提示用户)。

需求 5:导入流程是 选文件 → exifr 读 EXIF 信息(拍摄时间/设备/GPS 等) → 生成 256x256 缩略图压缩到质量 0.7(快速显示用) → 算 p-hash 感知哈希(64 位指纹,旋转缩放不变) → 查重,已存在的照片弹确认框(覆盖/跳过/都保留)。

需求 6:照片元数据(EXIF/缩略图/标签/相册)存 IndexedDB(支持大数据量),原图二进制存 OPFS(Origin Private File System,浏览器给的私有大文件存储,比 IndexedDB 存 blob 性能好),两者通过 Photo ID 关联。

需求 7:人脸检测用 face-api.js(浏览器里跑 TensorFlow.js 模型,纯前端),检测出来后用 dbscan 算法做聚类(基于人脸特征向量距离),未标名字的人脸分组显示"未命名 1"、"未命名 2",用户后续在分组页里标名字。

需求 8:浏览支持三种视图,网格视图(等大缩略图平铺,适合快速浏览)、瀑布流(不等高,适合竖构图多的相册)、时间线(按拍摄时间倒序,适合回忆)。

需求 9:大图查看支持左右翻页(键盘左右键)、鼠标滚轮缩放、旋转按钮(纠正 EXIF 方向错的照片)。

需求 10:标签多选维护自动字典(所有出现过的标签展示成可选),重命名标签时所有引用同步更新(用 tag_id 关联不用 string 比较),支持嵌套路径(如 "2024/旅行/日本")用树形结构。

需求 11:人脸分组单独一个页面,展示所有人物分组,点进去是某个人物下的所有照片,支持合并分组(把"未命名 1"和"未命名 2"手动合成一个)。

需求 12:批量操作(批量加标签/批量移相册/批量删除)用 Web Worker 跑(主线程只更新进度条),处理进度可视化,几万张照片不卡 UI。

技术栈:React 18 + Vite + TypeScript + CSS Modules；exifr(读 EXIF)+ heic2any(转 HEIC)+ face-api.js(人脸检测)；IndexedDB + OPFS 双存储；Vite 默认 5173 端口；纯前端不上传；useDB composable 封装所有 IndexedDB/OPFS 操作,业务组件不直接接触底层 API。
